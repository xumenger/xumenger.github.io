在深究Redis 源码之前，先对Redis 的特性进行简单列举

* Redis 使用RDB 存储文件，另外还支持AOF 存储
* Redis 是单进程单线程的服务端应用，基于Reactor 网络模型对外提供服务
* Redis 数据的备份是通过fork 出一个进程，原进程继续服务，fork 出来的进程将内存中的数据落地
* Redis 提供了键值对、哈希表、集合等大量的数据结构
 
在研究Redis 源码之前，带着这样几个疑问作为切入点

* Redis 服务端中对于各个客户端连接是怎么进行管理的？
* Redis 中为key-value 设置有效期，到了有效期后删除的逻辑是怎么实现的？
* 主从模式下，主从之间的数据是怎么交互的？
* Redis中字符串数据结构是怎么设计的？
* Redis中集合数据类型的交集、差集、并集是怎么实现的？
 
希望通过研究Redis 的源码深究网络编程、Redis 的数据结构、网络协议设计

阅读源码时，要先抓住系统的框架主线，弄清楚架构之后，再去深究源码细节！

## redis.h/redis.c
 
redis.h 定义了Redis 配置项的可选值、默认值等宏

定义了结构体redisClient，用于维护每个Redis 客户端的状态信息，方便服务端管理

定义了结构体redisServer，用于管理Redis 服务端的信息，这是一个庞大的结构体，包含大量的字段信息

redis.c 定义了一个命令表redisCommandTable，将get、set、setnx、setex、sadd 等Redis 命令对应的服务端函数一一对应，等到客户端执行get [key] 命令时，根据这个表就会再服务端调用对应的函数，这个和Kafka 中的KafkaApis 的设计类似

## ae.c 事件驱动相关

main() 函数定义在redis.c 这个源文件中，作为服务端的启动入口，调用initServer() 对redisServer 结构体的相关字段赋值

initServer() 中调用ae.c 中封装的aeCreateEventLoop() 函数创建一个事件驱动器。ae.c 事件驱动的实现有ae_evport.c、ae_epoll.c、ae_kqueue.c、ae_select.c，根据编译时候的宏选择使用哪个实现

* HAVE_EVPORT ==> ae_evport.c
* HAVE_EPOLL ==> ae_epoll.c
* HAVE_KQUEUE ==> ae_kqueue.c
* 其他 ==> ae_select.c

比如我们接下来选择ae_epoll.c 进行分析

aeCreateEventLoop() 内部调用ae_epoll.c 的aeApiCreate() 的方法，其内部实际是调用epoll_create(1024) 

epoll_create() 函数生成一个epoll专用的文件描述符。它其实是在内核申请一空间，用来存放你想关注的socket fd上是否发生以及发生了什么事件。size就是你在这个epoll fd上能关注的最大socket fd数

ae_epoll.c 里面对epoll 系列函数进行了封装！epoll 系列API 的知识扩展研究方向

* 水平模式 VS 边沿模式
* epoll 在操作系统底层是使用什么数据结构管理所有的socketFD 的？好像是红黑树

initServer() 函数中还调用了aeCreateFileEvent()，为服务端的socket 注册AE_READABLE 事件，监听是否有客户端连接到服务端
        
* aeCreateFileEvent() 调用时，对应添加的回调函数是acceptTcpHandler()，即当服务端socket 上有新的客户端连接进来后，会回调这个函数处理
* acceptTcpHandler() 定义在networking.c 源文件中
* acceptTcpHandler() 调用 createClient()，创建redisClient
* 注意在createClient() 中对于客户端socketfd 的参数的设置：NonBlock、EnableTcpNoDelay
* createClient() 同时调用aeCreateFileEvent() 为客户端socket 注册AE_READABLE 事件，aeCreateFileEvent(server.el, fd, AE_READABLE, readQueryFromClient, c)
* 特别注意aeCreateFileEvent() 设置的第四个参数：回调函数、第五个参数：客户端对象！！！

main() 函数接下来调用aeProcessEvents() 方法处理IO 事件，逐层调用到ae_epoll.c 中的aeApiPoll()，再调用到epoll_wait()

aeProcessEvents() 中拿到发生了IO 事件的aeFileEvent 对象，判断可读、可写事件是否发生，然后回调对应之前注册的方法

上面的流程看到了serverSocketFd 注册了可读时间、clientSocketFd 注册了可读时间，但是暂时还没有看到哪里注册了可写事件

## readQueryFromClient()

一直以来有一个问题，当某个clientSocketFd 上发生事件的时候，怎么知道对应是哪个redisClient

在为客户端socket 注册可读时间的时候。aeCreateFileEvent(server.el, fd, AE_READABLE, readQueryFromClient, c)，最后一个参数，就是对应的redisClient 的指针

当可读时间发生的时候，回调readQueryFromClient()，会将这个用户自定义指针也作为参数传进来

```c
/* 
 * networking.c 源文件
 * 回调第三个参数就是注册事件时传入的
 */
void readQueryFromClient(aeEventLoop *el, int fd, void *privdata, int mask) {
    redisClient *c = (redisClient*) privdata;
    int nread, readlen;
    size_t qblen;
    REDIS_NOTUSED(el);
    REDIS_NOTUSED(mask);

    server.current_client = c;
    readlen = REDIS_IOBUF_LEN;
    /* If this is a multi bulk request, and we are processing a bulk reply
     * that is large enough, try to maximize the probability that the query
     * buffer contains exactly the SDS string representing the object, even
     * at the risk of requiring more read(2) calls. This way the function
     * processMultiBulkBuffer() can avoid copying buffers to create the
     * Redis Object representing the argument. */
    if (c->reqtype == REDIS_REQ_MULTIBULK && c->multibulklen && c->bulklen != -1
        && c->bulklen >= REDIS_MBULK_BIG_ARG)
    {
        int remaining = (unsigned)(c->bulklen+2)-sdslen(c->querybuf);

        if (remaining < readlen) readlen = remaining;
    }

    qblen = sdslen(c->querybuf);
    if (c->querybuf_peak < qblen) c->querybuf_peak = qblen;
    c->querybuf = sdsMakeRoomFor(c->querybuf, readlen);
    nread = read(fd, c->querybuf+qblen, readlen);
    if (nread == -1) {
        if (errno == EAGAIN) {
            nread = 0;
        } else {
            redisLog(REDIS_VERBOSE, "Reading from client: %s",strerror(errno));
            freeClient(c);
            return;
        }
    } else if (nread == 0) {
        redisLog(REDIS_VERBOSE, "Client closed connection");
        freeClient(c);
        return;
    }
    if (nread) {
        sdsIncrLen(c->querybuf,nread);
        c->lastinteraction = server.unixtime;
        if (c->flags & REDIS_MASTER) c->reploff += nread;
    } else {
        server.current_client = NULL;
        return;
    }
    if (sdslen(c->querybuf) > server.client_max_querybuf_len) {
        sds ci = catClientInfoString(sdsempty(),c), bytes = sdsempty();

        bytes = sdscatrepr(bytes,c->querybuf,64);
        redisLog(REDIS_WARNING,"Closing client that reached max query buffer length: %s (qbuf initial bytes: %s)", ci, bytes);
        sdsfree(ci);
        sdsfree(bytes);
        freeClient(c);
        return;
    }
    processInputBuffer(c);
    server.current_client = NULL;
}
````

## 以get 命令为例

继续看readQueryFromClient()，以分析get 命令对应的处理流程作为切入点分析

read() 函数如果返回0，说明客户端调用了close()，服务端对应需要对于clientSocketFd 调用close()

readQueryFromClient()函数内部再调用processInputBuffer() 处理获取

processInputBuffer() 调用processInlineBuffer() 函数对于行格式的网络协议进行解析，然后调用processCommand() 函数处理这个客户端发过来的命令，这里面关于redisClient 内部使用什么数据结构存储网络包、怎么解析网络包的，内存是怎么管理的，这个值得好好研究！！！具体的数据结构封装在sds.c 这个源文件中

processCommand() 函数的实现在redis.c 中，调用lookupCommand() 根据客户端输入的命令类型，找到这个命令，命令和对应的函数，被存储在字典结构体中，相关的字典结构体封装在dict.c 这个源文件中（字典结构体在存储键值对等数据的时候也广泛使用，是Redis 中的一个核心数据结构）

找到这个命令对应的处理函数后，调用call()，然后通过redisClient->cmd->proc(c); 执行之前注册的函数，比如get 命令对应的处理函数是getCommand()。各个命令对应的处理函数在redis.c 源码的最开始定义在redisCommandTable 中

getCommand() 定义在t_string.c 中，逐层调用，会调用到lookupKeyReadOrReply() 从内存中根据键值查找值信息
        
lookupKeyReadOrReply() 定义在db.c 中，逐层调用，会调用到dictFind(db->dict,key->ptr); 可以看到每个db 对象中都有dict 属性去存储字典信息，具体会调用到dict.c 中的dictFind() 方法

dict.c 中封装了用C 实现的字典数据结构，存储键值对信息（后续继续研究，估计这个数据结构还会用在其他更多地方）

## dict.c 字典结构体

dict.c 相关的源码也就不到1000 行！

dict.c 中定义dictType 结构体，这个结构体有一个hashFunction 用于指定hash 函数

redis 中相关的dictType 与对应的hash 函数，都定义在redis.c 中

db 对象中的dict 使用的dictType 是dbDictType，对应的hash 函数是dictSdsHash()，在initServer() 初始化redisServer 的时候指定

dictSdsHash() 的定义如下，hash 函数的设计是实现Hash 表的一个重点！！

```c
// redis.c
unsigned int dictSdsHash(const void *key) {
    return dictGenHashFunction((unsigned char*)key, sdslen((char*)key));
}
 
// dict.c
unsigned int dictGenHashFunction(const void *key, int len) {
    /* 'm' and 'r' are mixing constants generated offline.
     They're not really 'magic', they just happen to work well.  */
    uint32_t seed = dict_hash_function_seed;    // 5381
    const uint32_t m = 0x5bd1e995;
    const int r = 24;
 
    /* Initialize the hash to a 'random' value */
    uint32_t h = seed ^ len;
 
    /* Mix 4 bytes at a time into the hash */
    const unsigned char *data = (const unsigned char *)key;
 
    while(len >= 4) {
        uint32_t k = *(uint32_t*)data;
 
        k *= m;
        k ^= k >> r;
        k *= m;
 
        h *= m;
        h ^= k;
 
        data += 4;
        len -= 4;
    }
 
    /* Handle the last few bytes of the input array  */
    switch(len) {
    case 3: h ^= data[2] << 16;
    case 2: h ^= data[1] << 8;
    case 1: h ^= data[0]; h *= m;
    };
 
    /* Do a few final mixes of the hash to ensure the last few
     * bytes are well-incorporated. */
    h ^= h >> 13;
    h *= m;
    h ^= h >> 15;
 
    return (unsigned int)h;
}
```
