## 在Spring 中集成RateLimiter

首先在pom.xml 中添加guava 依赖
 
```xml
<dependency>
  <groupId>com.google.guava</groupId>
  <artifactId>guava</artifactId>
  <version>18.0</version>
</dependency>
```
 
自定义限流注解，如果某个方法需要限流，则可以使用该注解修饰
 
```java
package com.xum.ratelimiter;
 
import java.lang.annotation.Documented;
import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;
 
/**
 * 自定义限流注解
 */
@Target({ElementType.PARAMETER, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface LimitAnno {
    // 描述
    String desc() default "";
    // key
    String key() default "";
}
```
 
实现一个限流的切面，在这里实现RateLimiter 的管理，以及对@LimitAnno 修饰了的函数的限流逻辑
 
```java
package com.xum.ratelimiter;
 
import java.lang.reflect.Method;
import java.util.concurrent.TimeUnit;
 
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.Pointcut;
import org.aspectj.lang.reflect.MethodSignature;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Configuration;
import org.springframework.core.annotation.Order;
import org.springframework.core.env.Environment;
 
import com.google.common.cache.CacheBuilder;
import com.google.common.cache.CacheLoader;
import com.google.common.cache.LoadingCache;
import com.google.common.util.concurrent.RateLimiter;
 
/**
 * 限流AOP
 */
@Aspect
@Configuration
@Order(1)
public class LimitAspect
{
    @Autowired
    private Environment env;
    
    LoadingCache<String, RateLimiter> caches = null;
 
    public LimitAspect() 
    {
        // 根据key 分不同的令牌桶，每天自动清理缓存
        // LoadingCache 用于管理所有的RateLimiter
        caches = CacheBuilder.newBuilder()
                            .maximumSize(1000)
                            .expireAfterWrite(1, TimeUnit.DAYS)
                            .build(new CacheLoader<String, RateLimiter>() {
                                @Override
                                public RateLimiter load(String key) {
                                     // 如果application.yml 中没有对应的配置，则使用默认值4
                                    Integer value = 4;
                                    try {
                                        // 使用 Environment env 从application.yml 中获取配置自定义配置信息
                                        String permits = env.getProperty("ratelimiter." + key);
                                         value = Integer.valueOf(permits);
                                    } catch (Exception e) {
                                       
                                    }
                                    // 新的key 初始化，每秒只发出n 个令牌
                                    return RateLimiter.create(value);
                                }
                             });
    }
    
    // Service 层切点，限流
    @Pointcut("@annotation(com.xum.ratelimiter.LimitAnno)")
    public void ControllerAspect() {
        
    }
    
    // Around
    @Around("ControllerAspect()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable
    {
        MethodSignature signature = (MethodSignature)joinPoint.getSignature();
        Method method = signature.getMethod();
        LimitAnno limitAnno = method.getAnnotation(LimitAnno.class);
        String key = limitAnno.key();
        String desc = limitAnno.desc();
        Object obj;
        try {
            // 从LoadingCache<String, RateLimiter> caches 根据key 获取其对应的RateLimiter
            RateLimiter rateLimiter = caches.get(key);
            Boolean flag = rateLimiter.tryAcquire();
            if (flag) {
                obj = joinPoint.proceed();
            } else {
                return desc + "-" + key + ": Request Too Fast!";
            }
        } catch (Throwable e) {
            throw e;
        }
        
        return obj;
    }
}
```
 
然后在对应的Controller 上添加限流注解即可
 
```java
package com.xum.controller;
 
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.ResponseBody;
import org.springframework.web.bind.annotation.RestController;
 
@RestController
@RequestMapping(path = "/test")
public class TestController 
{
    // ...
    
    @RequestMapping(value = "/add")
    @ResponseBody
    @LimitAnno(desc="测试HTTP接口", key="testAdd")
    public String add(@RequestBody String request) 
    {
        // 简单模拟一下直接返回
        return "Success!"
    }
 
}
```
 
在application.yml 中可以对应进行如下的配置
 
```yml
ratelimiter:
  testAdd: 5
```

## RateLimiter 源码分析

>[Guava限流器RateLimiter中mutexDoNotUseDirectly/锁的使用](https://blog.csdn.net/egg1996911/article/details/103928573)

RateLimiter 的源码，了解Guava 的令牌桶算法实现限流，下面是关于并发的关键逻辑
 
```java
// Can't be initialized in the constructor because mocks don't call the constructor.
private volatile Object mutexDoNotUseDirectly;
 
private Object mutex() {
    Object mutex = mutexDoNotUseDirectly;
    if (mutex == null) {
        synchronized (this) {
            mutex = mutexDoNotUseDirectly;
            if (mutex == null) {
                mutexDoNotUseDirectly = mutex = new Object();
            }
        }
    }
    return mutex;
}

public final void setRate(double permitsPerSecond) {
    checkArgument(permitsPerSecond > 0.0 && !Double.isNaN(permitsPerSecond), "rate must be positive");
    synchronized (mutex()) {
        doSetRate(permitsPerSecond, stopwatch.readMicros());
    }
}
 
final long reserve(int permits) {
    checkPermits(permits);
    synchronized (mutex()) {
        return reserveAndGetWaitLength(permits, stopwatch.readMicros());
    }
}
```

问题列表与总结：

* 为什么不直接用synchronized(this)呢？
  * 单独设立一个对象实例来加锁，可以在一个对象里存在多把不同的锁，让锁的力度更细
  * 此外，锁的是对象内部的实例，这可以避免对象外部的操作锁住对象实例本身而导致对象内部使用了synchronized (this)的行为都被影响（即与无关的行为共用了this这一把锁）
* 为什么RateLimiter这样使用并发同步机制。为什么不直接用饿汉模式初始化mutexDoNotUseDirectly呢？
  * [https://github.com/google/guava/issues/3066](https://github.com/google/guava/issues/3066)
  * 作者是考虑到使用Mockito框架时，用mock方法创建RateLimiter的mock对象，此时RateLimiter的构造函数（包括直接赋值）都不会被执行（大家可以自己试试，的确如此）
* 为什么这里不直接使用mutexDoNotUseDirectly，而是要额外声明一个局部变量mutex呢？
  * [https://github.com/google/guava/issues/3381](https://github.com/google/guava/issues/3381)
  * 不管是初始化情况下(从4次减少到3次)或者不需要初始化的情况(从2次减少到1次)下，都能减少volatile变量(mutexDoNotUseDirectly)读1次
  * 而volatile变量在缓存中失效时，cpu需要直接去访问内存中的最新值，访问内存的速度显然是不如访问cpu自身缓存来得快，因此使用volatile变量的读写比使用非volatile变量成本更高。
  * 所以使用局部变量mutex的目的就是为了减少volatile变量的读次数，从而提高效率！
  * 如何设计场景测试使用volatile在缓存失效时，cpu直接去访问内存中的最新值具体影响性能有严重？有没有测出这个数据指标
